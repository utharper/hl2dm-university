10000 frame limit in VirtualDub

Walking Target

2007-03-10 22:19:42

I am working on converting the FIOS demos. I had trouble because I could only import 10000 frames of footage into VirtualDub.

I posted on the vdub forums and the maker of the program informed me that HL2 might be exporting frames that are not in sequence after 10000 frames.

This proved to be the case. With host framerate 30 it exports frames starting at 300000. When it reaches 309999 and flips over, it doesnt go to 310000 like you would expect it to. Instead it adds another position and goes to 3010000. This causes vdub to think the sequence is over, so it ends the import there.

I was planning on removing the extra postion from the later frames, but then I realized that no matter how long your movies are, the first 10000 frames are always consistent, so it is more efficient to add the extra position to the first 10000 frames.

If anyone knows of a good tool to batch rename, please let me know.

Thanks!

Here it is:

Batch File Renamer 2.51

A simple little app. Just grab the first 10000 frames, move em to a folder, then
fire up this little program and use the "Catalog Files". Set the starting number to 3000000. This will renumber the first 10000 TGA files back into sequence.

Ok, so I looked at all the other xvid encoded movies out there. All of them get the slight distortion around the crosshairs and the text. Now I know why the Cyn vid looks so clean, they removed the crosshairs altogether, swtiched off chat and replaced the HUD with one added during the edit.

The best compromise I can come up with is to leave the original HUD, but make sure chat is switched off and use a single dot for crosshairs (or the smallest crosshairs possible). When watching the video most of the attention will naturally be drawn to the crosshairs, so you want as little distortion there as possible.

[GoTg] Walking Target wrote:Heh, we'll see how they finally turn out. I am still learning as I go. So far I am not very happy with the results, but I may just be too picky.
I've found If you go from a resolution such as 1280x1024 (what I use) and keep the same crosshair (assuming you have a single dot modified from the original file) as you change video size to 640x480 (what xvid has best compression/quality at) the dot is friggin huge.

To compensate, remove (to safe folder) your halflife2.ttf file and let the game generate a default 5 dot crosshair, then edit the 4 dots out leaving the center dot. This creates the correct size for 640x480 xhair single dot and not some enormous balloon for a dot.
